# 大白话解释

## 1. 这个项目到底做什么？

一句话：

```text
我们在研究：病人历史记录很长时，能不能先删掉低价值内容，只保留关键医疗信息，从而让 LLM 更省 token、更省内存、更快，同时还能答对问题。
```

这个项目不是训练一个新模型。

这个项目做的是：

```text
长病历上下文压缩 / context retention / prompt 压缩
```

我们把一个病人的长期病史整理成 timeline，例如：

```text
2018-01-01  diagnosed with diabetes
2018-02-01  started metformin
2018-03-01  routine follow-up, stable
2018-04-01  normal lab reviewed
2018-05-01  adverse event: hypoglycemia
2018-05-10  hospitalized for severe hypoglycemia
```

然后问问题：

```text
Why was the patient hospitalized?
```

LLM 要回答：

```text
severe hypoglycemia after hypoglycemia
```

关键问题是：

```text
如果病史很长，我们到底该把哪些记录放进 prompt？
```

这就是本项目的核心。

## 2. 我们为什么要做这个？

LLM 处理长上下文时，成本会变高：

```text
更多 input tokens
→ 更高 prefill latency
→ 更大 KV-cache / memory
→ edge device 上更难跑
```

课程是 Efficient Machine Learning，所以我们要证明：

```text
减少输入上下文，也是一种效率优化。
```

我们的目标不是单纯让它最快，而是找到更好的 trade-off：

```text
少 token + 保住关键证据 + 答案还准确
```

## 3. 我们比较了哪几种方法？

一共四种：

```text
1. full_context
2. sliding_window
3. selective
4. selective_query_aware
```

## 4. full_context 是什么？

最简单：

```text
把病人的完整 timeline 全部塞进 prompt。
```

优点：

```text
信息最全，最不容易漏证据。
```

缺点：

```text
token 最多，memory 最大，latency 最高。
```

它是我们的 baseline / upper bound。

## 5. sliding_window 是什么？

大白话：

```text
只看最近的病历，旧的全部不要。
```

例如只保留最后 50% 的 events：

```text
old diagnosis      丢掉
old medication     丢掉
recent follow-up   保留
recent normal lab  保留
```

优点：

```text
很快，token 少。
```

缺点：

```text
如果答案依赖早期关键事件，它会直接答错。
```

这就是为什么我们的实验里 sliding_window 虽然快，但 accuracy / evidence recall 很差。

## 6. selective 是什么？

这是我们的核心方法。

大白话：

```text
不是按新旧保留，而是按医学重要性保留。
```

我们会给每条 event 打分。

重要事件高分：

```text
diagnosis
medication_start
medication_change
medication_stop
adverse_event
hospitalization
```

普通噪声低分：

```text
routine follow-up
normal lab
stable status note
administrative note
```

然后保留分数最高的一部分。

例如：

```text
--retention-ratio 0.5
```

意思是：

```text
每个病人只保留约 50% 的 events。
```

这就是 selective retention。

## 7. selective_query_aware 是什么？

这是 selective 的增强版。

大白话：

```text
不只看医学重要性，还看这条记录和当前问题有没有关系。
```

例如问题是：

```text
Which medication was stopped?
```

那么 medication 相关 events 会加分。

如果问题是：

```text
Why was the patient hospitalized?
```

那么 hospitalization / adverse event 相关 events 会加分。

## 8. 我们的创新点到底在哪里？

不要说：

```text
我们就是用了 top-k。
```

这样听起来太简单。

要说：

```text
我们提出了一个 selective retention framework。
```

完整逻辑是：

```text
patient timeline
→ event-level importance scoring
→ retention ratio control
→ keep high-value events
→ reduce prompt tokens
→ run QA inference
→ measure accuracy / latency / memory
```

Top-k 只是最后一步。

真正的创新是：

```text
在 tokenization 之前，先根据医疗事件重要性筛选上下文。
```

## 9. 当前项目用的是什么数据？

默认用 repo 自己生成的 synthetic EHR。

也就是：

```text
不是现实病人数据
不是 MIMIC
不是受限医疗数据
```

我们自己生成病人的长期医疗记录，包括：

```text
diagnosis
medication
procedure
abnormal lab
adverse event
hospitalization
routine follow-up
normal lab
```

这样做的好处：

```text
可复现
没有隐私问题
不用申请数据权限
能精确知道 gold evidence 在哪里
```

## 10. 为什么 synthetic 数据可以用？

因为我们的重点不是证明医学结论。

我们的重点是证明：

```text
当长上下文里有关键事件和大量噪声时，
selective retention 是否比 sliding window 更能保住关键证据。
```

synthetic 数据正好可以控制这些因素。

报告里可以诚实写：

```text
This is a controlled synthetic longitudinal EHR evaluation.
```

不要吹成：

```text
真实临床验证。
```

## 11. 当前跑出来的结果怎么理解？

当前正式设置：

```text
300 patients
900 QA samples
18,600 events
high noise
long timeline
```

大概结果是：

```text
full_context:
准确，但 token 最多

sliding_window:
最快，但把关键旧证据删掉，所以答错

selective:
token 减少约 50%，但关键证据还在，所以还能答对

selective_query_aware:
和 selective 类似，多了 query relevance
```

论文里最重要的结论：

```text
Sliding window is faster but loses critical evidence.
Selective retention achieves a better quality-efficiency trade-off.
```

## 13. 每个核心文件是干什么的？

```text

```

生成 synthetic 病人事件。

```text
src/ehr_retention/timeline.py
```

把 events 按 patient 和 date 整理成 timeline。

```text
src/ehr_retention/qa_generation.py
```

根据 timeline 自动生成 QA 问题和标准答案。

```text
src/ehr_retention/retention.py
```

最核心文件。四种 retention strategy 的规则都在这里。

```text
src/ehr_retention/inference.py
```

推理 backend。默认是 rule_local，以后可以接 LLM。

```text
src/ehr_retention/metrics.py
```

算 exact match、F1、evidence recall。

```text
scripts/run_experiment.py
```

真正跑实验的入口。

## 14. 量化组员怎么配合？

量化组员不需要改数据生成部分。

他只需要复用这两个文件：

```text
data/processed/timelines.jsonl
data/processed/qa.jsonl
```

然后对同一批 QA 跑不同模型：

```text
FP16 model
INT8 model
INT4 model
```

他要保证：

```text
同一批 question
同一批 retained context
同一套 prompt
只改变模型压缩方式
```

这样才公平。

最理想的整体实验矩阵：

```text
Context strategy:
full_context / sliding_window / selective

Model setting:
FP16 / INT8 / INT4
```

最终可以得到：

```text
full_context + FP16
full_context + INT8
sliding_window + FP16
sliding_window + INT8
selective + FP16
selective + INT8
selective + INT4
```

## 16. 如果换成别的数据，怎么复用？

核心要求：

```text
把新数据转成我们项目的 timeline schema。
```

每条 event 至少要有：

```json
{
  "patient_id": "P00001",
  "event_id": "P00001-E0001",
  "date": "2018-01-01",
  "event_type": "diagnosis",
  "text": "Diagnosed with type 2 diabetes.",
  "value": "type 2 diabetes",
  "importance": "high"
}
```

然后生成 timeline：

```json
{
  "patient_id": "P00001",
  "events": [...],
  "timeline_text": "..."
}
```

QA 文件需要：

```json
{
  "sample_id": "P00001-DX",
  "patient_id": "P00001",
  "task_type": "diagnosis_history",
  "question": "What diagnosis was first recorded?",
  "answer": "type 2 diabetes on 2018-01-01",
  "gold_evidence_event_ids": ["P00001-E0001"]
}
```

只要新数据能转成这两个文件：

```text
timelines.jsonl
qa.jsonl
```

后面的 retention、experiment、evaluation 都不用大改。

Synthea 是最适合的外部 synthetic 数据。

可以把 CSV 放到：

```text

```

例如：

```text

medications.csv
procedures.csv

```

```text
src/ehr_retention/synthea_adapter.py
```

这个 adapter 的作用是：

```text

```

如果后面真的要完整接 Synthea，只需要补一个脚本：

```text
scripts/prepare_synthea.py
```

## 19. 哪些结果可以写进论文？

可以写：

```text
Selective retention reduces input tokens by about 50% while preserving gold evidence in the controlled synthetic EHR benchmark.
```

可以写：

```text
Sliding window is efficient but fails when older clinical evidence is needed.
```

可以写：

```text
The method is compatible with quantization because it reduces input-side context while quantization reduces model-side memory.
```

不要写：

```text
We clinically validated the method on real patient records.
```

也不要写：

```text
Our method is always better than sliding window.
```

更准确的说法是：

```text
Our method improves the quality-efficiency trade-off when clinically important evidence is sparse and may appear far back in the patient timeline.
```

## 20. 这个项目和量化那一半怎么合并成完整 final project？

最终大框架应该是：

```text
Input-side efficiency:
selective retention

Model-side efficiency:
quantization
```

合起来就是：

```text
memory-efficient healthcare LLM on edge device
```

报告结构可以这样写：

```text
1. Problem:
Longitudinal healthcare QA has long context and edge memory constraints.

2. Method:
Selective retention reduces input tokens.
Quantization reduces model memory.

3. Baselines:
Full context.
Sliding window.
Unquantized model.

4. Metrics:
Accuracy / F1 / evidence recall.
Input tokens.
Latency.
Peak memory.

5. Results:
Selective retention preserves evidence with fewer tokens.
Quantization further reduces model memory.
Combined method gives best deployment trade-off.
```

## 21. 最后一句话总结

这个项目不是在证明一个复杂医学模型。

它在证明：

```text
在长程医疗 QA 中，直接保留最近窗口不可靠；
用医学重要性做 selective retention，可以在减少 token 的同时保留关键证据；
这和量化结合后，能形成一套更适合 edge device 的 efficient ML pipeline。
```
